# YDS Command Centre — Technical Workspace

You are a **build and engineering assistant** for the YDS Command Centre — a web-based interface for Colin (AI Chief of Staff) at YourDesignStore.

This is a **separate engineering repo**. Colin's business workspace lives at `../dan/`. This repo contains only the web application code.

---

## What This App Does

The Command Centre gives Dan (CEO) a browser-based interface to:
- **Chat with Colin** — streaming AI conversation with tool use (Notion queries, file writes)
- **Trigger skills** — `/brief`, `/decide`, `/dump`, `/health`, `/review`, `/route`
- **Browse Notion** — all 7 databases, any page, full content rendered as markdown
- **View dashboard** — Focus Areas health, overdue Commitments, recent Decisions
- **Read documents** — briefings, decisions, weekly reviews generated by Colin

---

## Architecture

```
Browser (Alpine.js)  ←— SSE —→  Express Server  ←— SDK —→  Claude API (Opus)
                                       |
                                  Tool Handler
                                 /     |     \
                            Notion   Files   Approval Gate
```

**Stack:**
- Node.js + Express (no framework)
- Anthropic Claude SDK (`@anthropic-ai/sdk`) — streaming chat with tool use
- Notion SDK (`@notionhq/client`) — database queries, page content
- Alpine.js — frontend reactivity (no build step)
- marked.js — markdown rendering
- SSE — real-time streaming responses

**No build tooling.** Edit HTML/JS/CSS, refresh browser.

## Coding Conventions

### Backend
- CommonJS modules (`require`/`module.exports`) — no ES modules
- Express route handlers in `server/routes/`
- Business logic in `server/services/`
- Tool definitions in `server/tools/`
- All Notion API calls go through `server/services/notion.js` (5-min cache)
- SSE streaming for chat responses (event types: `text`, `tool_use`, `approval`, `error`, `done`)

### Frontend
- **Alpine.js** for all reactivity — `x-data`, `x-show`, `x-for`, `x-text`, `x-html`
- Single `app()` function in `public/js/app.js` — all state and methods
- **No framework, no components, no build step** — edit and refresh
- `marked.parse()` for rendering markdown content
- Dark theme with CSS custom properties (defined in `:root`)

### Adding a New Feature

**New backend route:**
1. Create handler in `server/routes/yourroute.js`
2. Mount in `server.js`: `app.use('/api/yourroute', require('./server/routes/yourroute'))`

**New tool for Colin:**
1. Define schema in `server/tools/notion-tools.js` or `server/tools/file-tools.js`
2. Add execution logic in `server/tools/tool-handler.js`
3. If it's a write operation, add to the write tools Set (triggers approval gate)

**New frontend view:**
1. Add nav button in `public/index.html`
2. Add view template with `x-show="view === 'yourview'"`
3. Add state variables and methods in `public/js/app.js`
4. Add styles in `public/css/styles.css`

## Critical Patterns

### Approval Gate
Write operations (Notion create/update, file writes) MUST go through the approval system. The agent loop pauses, sends an SSE `approval` event, and waits for Dan to approve or reject. Never bypass this.

### Agent Loop
The chat endpoint runs an agentic loop: Claude responds → check for tool calls → execute or request approval → feed results back → repeat until no more tool calls. Do not break this loop pattern.

### Notion Caching
All Notion reads are cached for 5 minutes. When debugging stale data, use `POST /api/notion/cache/clear`.
